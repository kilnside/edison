# User Testing in Design Methodologies: Where Edison Sits

## Comparison Table

| Dimension | GV Design Sprint | IDEO Human-Centered Design | Double Diamond | Edison |
|-----------|-----------------|---------------------------|----------------|--------|
| **Total duration** | 5 days | Weeks to months | Weeks to months | 20-40 minutes |
| **When user contact happens** | Day 5 (after prototype) | Phase 1: Empathize (before ideation) | Diamond 1: Discover + Diamond 2: Deliver | Never (research proxies only) |
| **What precedes user contact** | Map, Sketch, Decide, Prototype (Days 1-4) | Problem framing | Problem definition | N/A |
| **What follows user contact** | Iteration or ship decision | Ideation, Prototyping | Development, Delivery | N/A |
| **Type of user contact** | Moderated 1:1 interviews with prototype | Ethnographic observation, interviews | Surveys, interviews, co-design | Proxy signals from forums, reviews, issues |
| **Number of users** | 5 | Varies (often 5-15) | Varies | 0 (direct), many (indirect via proxy) |
| **Purpose of user contact** | Validate prototype decisions | Understand unmet needs | Validate problem and solution | Inform research, not validate |
| **Can it run without users?** | No (Day 5 is mandatory) | No (Empathize is the foundation) | Technically yes, but considered incomplete | Yes (designed for it) |

## Key Observation

The methodologies that include user testing place it in one of two positions:

1. **Before ideation** (IDEO Empathize, Double Diamond Discover) — to understand the problem space
2. **After prototyping** (GV Sprint Day 5, Double Diamond Deliver) — to validate a tangible solution

Edison occupies the space between ideation and prototyping — the decision phase. No established methodology places direct user testing here. This is not because the decision phase is unimportant, but because:

- Before ideation, you need user contact to understand the *problem* (you do not know what to ask about yet)
- After prototyping, you need user contact to validate the *solution* (you have something to show)
- During the decision phase, you need *research about users* rather than direct user contact, because there is nothing tangible to test and the problem is already framed

Edison's position in the workflow naturally calls for proxy user signals (what did users of similar things say?), not direct user testing (what do users think of this thing?). The "thing" does not exist yet.

## What This Means for P8

The methodological gap is smaller than it appears. Edison does not skip user testing — it operates in a phase where user testing is not the appropriate method. The appropriate method is user signal research: understanding user needs, complaints, and behaviors through existing data.

The real gap is that Edison does not formalize this. The research phase should explicitly require proxy user signal gathering as a category, and the final spec should identify what assumptions need direct user validation once a prototype exists.

This reframes P8 from "Edison lacks user testing" to "Edison should formalize proxy user signals in research and bridge to post-prototype validation."
